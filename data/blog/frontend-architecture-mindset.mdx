## 前端架構的真正問題

我之前遇過想要引入新的技術，聽到的說法是：因為舊的技術太難維護了。

然後在技術選型時，開始糾結要使用 Zustand 還是 Redux？要用 React Query 還是 SWR？

但這些都不是架構問題的核心。而且我個人覺得，現在有了 AI，很多的技術問題，都可以問到得到蠻全面的答案。

架構的核心問題是：**這個系統在 1 年後，還能讓不同的人安全地修改嗎？**

## 我在企業系統中看到的三個主要問題

### 1. 元件責任不清楚

最常見的問題是：一個元件同時處理 UI 渲染、API 呼叫、業務邏輯與狀態管理。

一開始一定寫很快，所有東西都在同一個檔案裡面，code 好找也好改，但是，當業務開始不斷的添加，相同的邏輯因為上下文的不同的輸入與輸出，開始 copy paste，

某一天突然收到有個邏輯要調一下，又或者後端要改一下 api 接口，

接著就要茫茫大海用全站搜尋，結果某個地方用佔位符號寫一下，bug 就出來了。

所以，基本上：

- **UI 元件**：只負責渲染
- **Container 元件**：負責資料獲取與狀態協調
- **業務邏輯 hooks**：把可複用的邏輯抽離成 custom hook
- **API 層**：所有請求集中管理，不散落在元件內

### 2. 沒有可預期的資料流

大型系統中，資料流混亂是最難維護的問題之一。

我傾向在專案初期就建立明確的資料流規範：

- Server state（API 資料）vs Client state（UI 狀態）要搞清楚，到底現在的資料是在 server 端處理，還是在 client 端處理
- 跨模組共享的資料要有明確的 owner，資料是屬於哪個模組的，就應該由該模組來管理
- 避免 prop drilling，但也避免過度依賴全局狀態

### 3. 模組邊界不清楚

有共用，但也要有邊界，將不同的業務模組分離出來，特別是在 monorepo 或多頁面應用中，模組之間的依賴關係如果不清楚，重構成本會非常高。

在每個功能模組下維護自己的：

- 元件
- hooks
- API service
- 型別定義
- 常數

讓模組可以獨立理解、獨立測試、獨立部署。

## 結論

企業級前端架構的核心不是選最新的技術，而是建立一個可以被團隊持續維護的系統。

這需要：清楚的元件責任、可預期的資料流，以及清楚的模組邊界。
