## 前端架構的真正問題

很多前端工程師在討論架構時，容易陷入工具選型的爭論：要用 Zustand 還是 Redux？要用 React Query 還是 SWR？

這些都不是架構問題的核心。

架構的核心問題是：**這個系統在 1 年後，還能讓不同的人安全地修改嗎？**

## 我在企業系統中看到的三個主要問題

### 1. 元件責任不清楚

最常見的問題是：一個元件同時處理 UI 渲染、API 呼叫、業務邏輯與狀態管理。

短期可以跑，但隨著業務複雜度提升，它會成為整個系統最危險的地方。

我的做法是明確拆分：
- **UI 元件**：只負責渲染，props in、events out
- **Container 元件**：負責資料獲取與狀態協調
- **業務邏輯 hooks**：把可複用的邏輯抽離成 custom hook
- **API 層**：所有請求集中管理，不散落在元件內

### 2. 沒有可預期的資料流

大型系統中，資料流混亂是最難維護的問題之一。

我傾向在專案初期就建立明確的資料流規範：
- Server state（API 資料）vs Client state（UI 狀態）清楚分開
- 跨模組共享的資料要有明確的 owner
- 避免 prop drilling，但也避免過度依賴全局狀態

### 3. 模組邊界不清楚

特別是在 monorepo 或多頁面應用中，模組之間的依賴關係如果不清楚，重構成本會非常高。

我習慣在每個功能模組下維護自己的：
- 元件
- hooks
- API service
- 型別定義
- 常數

讓模組可以獨立理解、獨立測試、獨立部署。

## 結論

企業級前端架構的核心不是選最新的技術，而是建立一個可以被團隊持續維護的系統。

這需要：清楚的元件責任、可預期的資料流，以及清楚的模組邊界。
