## AI 工具的三種使用層次

大多數工程師使用 AI 工具的方式，大概可以分成三個層次：

1. **問答層**：把 AI 當 Stack Overflow，問語法、問 API 用法
2. **輔助層**：用 AI 幫忙補全、解釋程式碼、做簡單重構
3. **流程層**：把 AI 整合進工程流程，讓它參與架構設計、spec 撰寫、code review、文件生成

大多數人停在前兩層。我也還在努力嘗試第三層。

## 我實際在用什麼

### GitHub Copilot

主要用在日常 coding。最有價值的不是自動補全，而是在我知道要做什麼、但不想打重複內容時幫我加速。

我現在會用它來跟我一起編輯 prompt 內容，貼上來自不同的需求規格和 api 介面，透過自動補全用來補全我想寫的 prompt。

透過定義好的 AI Instruction 和 Skill 描述，直接讓 Agent 除了檔案樣板，邏輯規範等能夠寫好了。

對於想了解專案中部分的邏輯，透過 github copilot 更是方便，可以快速的了解程式碼的邏輯。

對了，Github Copilot 還是性價比相對很高的輔助工具，可以對接新的 LLM 模型，支持的速度也很快。

### Claude Code / Codex CLI

用在比較大範圍，獨立的任務：

- 導入特定的技術，像是套用 open telemetry，建立分散式追蹤
- 依照設計文件產出初稿，產生新的業務頁面
- 大範圍重構時的初版草稿
- 有點類似 vibe coding, 當預期的完成的邊界明確時，可以異步進行，讓 cli 執行，不打斷工作節奏。

### CLAUDE.md / SKILL 文件

這是我覺得最有系統價值的做法。

在每個專案根目錄放一份 `CLAUDE.md`，定義這個專案的：

- 技術棧與架構原則
- 元件命名與資料夾規範
- 常見 pattern 的範例
- 特定業務邏輯的說明

這樣不管是換人接手，還是給 AI agent 理解專案，都能快速建立上下文。

### MCP（Model Context Protocol）

當還是有團隊還是有開發流程時，需求、設計、程式碼、測試、文件、部署等，這些資訊分散在不同的工具中，

透過 MCP 可以讓 AI 存取這些資訊，達到更自動與更準確的開發。

像是我們有 Figma 設計稿，透過 MCP 可以去讀 Figma，可以快速的產生初版程式碼，並且大幅的提高客製化的 UI，在實作上的準確度。

## 我學到的幾個原則

**AI 不能替代架構判斷。** 它很擅長產出初版草稿，但最終的架構決策還是要由工程師來做。

**提示品質決定輸出品質。** 給 AI 越清楚的上下文，它的輸出越有用。這也是為什麼我花很多時間在 instruction 設計上。

**把 AI 當協作者，不是負責人。** 最好的使用方式是：我定義問題、AI 提供方案、我做判斷。
