## AI 工具的三種使用層次

我觀察大多數工程師使用 AI 工具的方式，大概可以分成三個層次：

1. **問答層**：把 AI 當 Stack Overflow，問語法、問 API 用法
2. **輔助層**：用 AI 幫忙補全、解釋程式碼、做簡單重構
3. **流程層**：把 AI 整合進工程流程，讓它參與架構設計、spec 撰寫、code review、文件生成

大多數人停在前兩層。我這幾年一直在嘗試第三層。

## 我實際在用什麼

### GitHub Copilot
主要用在日常 coding。最有價值的不是自動補全，而是在我知道要做什麼、但不想打重複內容時幫我加速。

### Claude Code / Codex CLI
用在比較需要理解脈絡的任務：
- 理解一段複雜的既有程式碼
- 依照設計文件產出初稿
- 大範圍重構時的初版草稿

### CLAUDE.md / SKILL 文件
這是我覺得最有系統價值的做法。

在每個專案根目錄放一份 `CLAUDE.md`，定義這個專案的：
- 技術棧與架構原則
- 元件命名與資料夾規範
- 常見 pattern 的範例
- 特定業務邏輯的說明

這樣不管是換人接手，還是給 AI agent 理解專案，都能快速建立上下文。

### MCP（Model Context Protocol）
目前還在早期階段，但我認為 MCP 是讓 AI 真正「懂你的系統」的基礎設施。

透過 MCP，AI 不只能讀懂程式碼，還能連結到你的文件系統、API 規格、甚至操作你的工具。

## 我學到的幾個原則

**AI 不能替代架構判斷。** 它很擅長產出初版草稿，但最終的架構決策還是要由工程師來做。

**提示品質決定輸出品質。** 給 AI 越清楚的上下文，它的輸出越有用。這也是為什麼我花很多時間在 instruction 設計上。

**把 AI 當協作者，不是執行者。** 最好的使用方式是：我定義問題、AI 提供方案、我做判斷。
