## Three Levels of AI Usage

Most engineers tend to use AI tools in roughly three levels:

1. **Q&A level**: treating AI like an interactive Stack Overflow — asking about syntax and API usage
2. **Assist level**: using AI for autocomplete, code explanations, or simple refactors
3. **Workflow level**: integrating AI into the engineering process — architecture design, spec writing, code review, documentation

Most engineers stay in the first two levels. I am still working on getting further into the third.

## What I Actually Use

### GitHub Copilot

I mainly use Copilot during day-to-day coding. The biggest value is not autocomplete itself — it is speeding up the parts I already understand but do not want to retype by hand.

These days I also use it to help me edit prompts. I paste in requirements from different specs and API interfaces, and let autocomplete fill in what I want the prompt to say.

Once I have defined a set of AI instructions and skill descriptions, I can let an agent handle things like file templates and logic conventions directly.

For understanding the logic of a specific part of the codebase, Copilot is especially useful — it helps me get up to speed quickly without digging manually.

Also worth noting: Copilot still has a solid price-to-value ratio. It keeps up with new LLM models and the pace of support is fast.

### Claude Code / Codex CLI

I use these for larger, more self-contained tasks:

- Introducing specific technology — for example, adding OpenTelemetry to set up distributed tracing
- Generating a first draft from a design document, creating new business pages
- Preparing the first pass of a broader refactor
- Something like vibe coding — when the expected boundary is clear, I can hand it off and let the CLI run asynchronously without interrupting my flow

### CLAUDE.md and skill files

This is the practice I find most structurally valuable.

I put a `CLAUDE.md` at the root of each project, which defines:

- the tech stack and architecture principles
- component naming and folder conventions
- examples of common patterns
- notes on specific business logic

Whether a new person is picking up the project or an AI agent is trying to understand it, this file helps build context quickly.

### MCP (Model Context Protocol)

On a real team with a real development process, requirements, design, code, tests, documentation, and deployments are spread across many different tools.

MCP gives AI a way to access that information, which leads to more automated and more accurate development.

For example: we have Figma designs. Through MCP, AI can read Figma directly — which speeds up generating the initial code and significantly improves implementation accuracy for custom UI.

## Principles I Keep Coming Back To

**AI does not replace architectural judgment.** It is good at producing a first draft, but the final call on what is safe, maintainable, and worth keeping still belongs to the engineer.

**Context quality shapes output quality.** The more clearly you define the context, the more useful the output. That is why I put a lot of time into instruction design.

**Treat AI as a collaborator, not the person in charge.** I define the problem, AI helps explore or draft, and I make the final engineering decision.
